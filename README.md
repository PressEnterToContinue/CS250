CS 250: Software Development Lifecycle

This is the Sprint Review and Retrospective from my final project in CS 250, where the goal was to apply Agile and Scrum practices on the SNHU Travel application. Below are a few important reflections.

Interpreting user needs and user stories:
You can't build the right thing if you're guessing at what the user wants. User stories solve that by framing the need from the user's perspective and pairing it with acceptance criterias that define what "done" is supposed to look like. On this project, those criteria translated almost directly into testable steps, which helped keep the team align and reduced wasted effort.

My approach to development and the Agile processes I'll keep:
I've moved toward building incrementally instead of locking everything in up front. Small working pieces give you something to test and adjust as you go. The habits I want to keep going forward are sprint plannin, daily check-ins, and a Kanban board for at-a-glance status. I also learned to request the specific information I need rather than operate on assumptions, which helped saved rework efforts when the project pivoted mid-development.

Being a good team member:
It comes down to communication. The moments that worked were the ones where someone spoke up clearly and kept the right people in the loop. For example, emailing both the Product Owner and tester when the content focus changes so everyone shared the same picture. Each Scrum role is distinct, but the whole thing only works when people stay reliable, follow through, and share accountability.
